Animal Video Studio — 用户体验痛点分析

基于当前代码实现（server.js + public/app.js + lib/*）的根因分析

核心问题概览

当前系统有 3 个严重痛点、2 个高优先级问题，全部源自同一根因：innerHTML 全量替换 + 2.5s 轮询的渲染架构。这不是多个独立 bug，是一个架构问题的多个症状。

痛点一：页面疯狂刷新 / 图片闪烁

用户感知：页面不停闪烁、图片跳动、体验极不稳定

根因

app.js:666-675 — 每 2.5 秒执行一次 setInterval，调用 loadHistory() 触发全量 DOM 重建。renderTask() 使用 element.innerHTML = ... 把整个 tab 内容替换掉。
触发源频率影响范围
startPolling()每 2.5s首页列表 + 任务页（条件触发）
runTaskAction()每次点击生成按钮连续两次渲染：renderTask() → loadHistory()
hashchange 事件路由变化时全量重绘

具体症状

症状触发代码严重度
图片区域闪烁innerHTML 替换整个 asset/shot 卡片严重
滚动位置跳动DOM 节点被销毁再重建，浏览器无法保持滚动严重
操作反馈丢失按钮点击后 DOM 重建，hover/click 状态消失高

痛点二：提示词框瞬间关闭 / 无法阅读

用户感知：打开提示词 <details> 后马上又合上，根本看不清内容

根因

app.js:404（Assets）和 app.js:487-500（Shots）— 提示词包裹在 <details> 元素中，但每次渲染都是 innerHTML 重建。2.5s 轮询触发后，所有 <details> 回到默认关闭状态。 
用户操作流程：
1. 点击 <details> 展开提示词
2. ...2.5秒后轮询触发...
3. innerHTML = renderAssets()  →  整个卡片重建
4. <details> 回到 collapsed 状态 → 提示词消失
5. 用户再次点击展开
6. ...2.5秒后又关闭...  ∞ 循环

涉及位置

文件行号元素
app.js404Asset 卡片中的 prompt <details>
app.js489Shot 卡片中的 firstFramePrompt <details>
app.js499Shot 卡片中的 videoPrompt <details>
方向：要么改为增量更新（只更新状态变化的 DOM 节点），要么记住 <details> 的展开状态并在重新渲染后恢复。

痛点三：无手动操作入口 / 系统卡死时零退路

用户感知：系统出问题时完全无法手动介入，所有流程被锁死在自动化管道里

当前自动化管道（不可中断）

方向设定 → 生成脚本 → 审批脚本 → 生成基础资产 → 审批资产 → 生成构图/首帧 → 审批首帧 → 生成视频

每一步都依赖上一步的 审批状态，且没有旁路。

缺失的手动能力

缺失能力用户想要的操作当前状态
编辑提示词 手动修改 AI 生成的提示词后再点生成 缺失 提示词是只读的 <p class="prompt">，无 textarea
手动上传生成结果 复制提示词 → 去其他地方生成 → 上传图片回系统 缺失 无上传入口，无"手动完成"按钮
取消正在进行的生成 生成卡住时能取消重试 缺失 无取消 API，无取消按钮
跳过步骤 某一步已经手动完成，直接标记为审批通过 缺失 只能等 API 返回才能推进
复制提示词 + 引用图 一键复制提示词和关联图片，方便去外部工具使用 部分 有复制按钮，但不附带引用图片

为什么这是个严重问题

  • 即梦 API 可能不稳定、超时、返回质量差的图片
  • 10 分钟冷却期（server.js 中硬编码）意味着失败后要等很久才能重试
  • 没有任何 "fallback 模式" — 系统一旦出问题，用户只能干等
方向：增加"手动模式"—— 可编辑提示词 textarea + 上传按钮 + 手动标记审批通过。让系统从"全自动管道"变成"自动化 + 可手动介入"的混合模式。

痛点四：次要但影响体验的问题

高 双击渲染

app.js:598-599 — runTaskAction() 在点击生成按钮后连续调用 renderTask() 和 loadHistory()，导致页面内容快速闪变两次。

中 10 分钟冷却无反馈

server.js 中硬编码 10 分钟冷却。前端只显示 "running" 状态，没有倒计时，没有预计剩余时间。用户不知道还要等多久。

中 无进度指示

生成请求发出后，setImmediate() 在后台执行。前端没有任何进度条或状态指示，用户不知道是在排队、处理中、还是卡死了。

低 设置页输入框可能丢失焦点

如果用户在 Settings tab 编辑时恰好触发轮询，虽然 Settings 面板目前不重绘，但 tab 切换时可能状态不一致。

总结：问题优先级矩阵

优先级问题影响修复复杂度
P0页面闪烁 / <details> 自动关闭所有用户，每次使用中 — 需要改渲染架构
P0无手动操作入口系统异常时完全卡死高 — 需要新 API + UI
P1提示词不可编辑无法调整生成质量中 — textarea + 提交 API
P1双击渲染操作后画面闪烁低 — 删一行代码
P2无上传入口无法手动补全流程中 — 上传 API + UI
P2无取消/进度指示等待焦虑低-中

核心架构问题

P0 级别的前两个问题都指向同一个根因：全量 innerHTML 替换渲染。每次轮询或操作都把整个 DOM 子树销毁重建，导致：

  • 视觉闪烁（DOM 节点销毁→重建的间隙）
  • 状态丢失（<details> 展开、输入框焦点、滚动位置）
  • 用户交互被打断（点击后 DOM 重建导致事件目标消失）

建议方向：改为增量 DOM 更新。只更新状态真正变化的节点（如状态 badge、图片 URL），保留其余 DOM 不动。这样既能保持轮询的实时性，又不会引发闪烁和状态丢失。
